# Поиск аномальных респондентов в активности поисковых запросов SoS

Полусеместровый контроль №4. Алгоритм находит **аномально высокую активность
отдельных респондентов по отдельным брендам** и помечает респондент-дни для
очистки отчётной выборки SoS.

## Запуск одной командой

```bash
pip install -r requirements.txt
python solution_Белоусов_Марк_ШЦТ-111.py
```

Опционально:

```bash
python solution_Белоусов_Марк_ШЦТ-111.py --data data_train --output output   # свои пути
python solution_Белоусов_Марк_ШЦТ-111.py --analytics                          # доп. аналитика (§8.2)
```

Программа создаёт папку `output/`. Полный прогон — секунды (лимит 5 минут).
Алгоритм **детерминирован**: повторный запуск на тех же данных даёт тот же
`anomalies.csv` (случайность не используется).

## Обязательные выходные файлы

## Единицы анализа и удаления 

- **Триггер аномалии**: `(SubjectID, researchdate, BrandID, CategoryDelivery)` —
  объект, на котором обнаруживается чрезмерная активность.
- **Единица удаления**: `(SubjectID, researchdate)` — попадает в `anomalies.csv`.
  При пересчёте показателей удаляются **все** строки респондента за этот день.

## Агрегация OTS 

Анализируются только строки с `BrandinDelivery = 1` и непустым `CategoryDelivery`.
Для респондента *i*, бренда *j*, дня *k*:

```
daily_ots(i, j, k) = Weight(i, k) * count_rows(i, j, k)
```

(мини-пример задания: `Weight=5`, 3 строки → OTS = 15).

## Как устроен алгоритм

### 1. Робастный score высокого OTS (на уровне бренда)

Для каждой пары *(CategoryDelivery, BrandID)* строим распределение `log1p(daily_ots)`
по респондент-дням и считаем **модифицированный z-score** (Iglewicz & Hoaglin, 1993):

```
score = 0.6745 * (log1p(daily_ots) - median) / MAD
```

`median` и `MAD` (median absolute deviation) — робастные оценки: их не «утягивают»
сами выбросы, поэтому аномально высокий OTS получает большой `score`, а типичный —
малый. Лог-шкала — потому что OTS сильно скошен. Score считается **относительно
бренда**: высокий абсолютный OTS может быть нормой для популярного бренда, но
аномалией для нишевого. На данных ~90% триггеров оцениваются именно на бренд-уровне.

### 2. Иерархический эталон (устойчивость к редким брендам, §6)

Эталон распределения выбирается иерархически:

```
бренд (если наблюдений >= 30 и MAD > 0)  ->  CategoryDelivery  ->  вся выборка
```

Уровень эталона записывается в `reason` файла `anomaly_reasons.csv`.

### 3. Условие активности (data-driven, без хардкода)

Высокий OTS из-за **одного** запроса крупновесного респондента — это не аномальная
активность, а просто большой панельный вес. Поэтому требуем реально **повторную**
активность: `count_rows` должен превышать **верхний фенс Тьюки** (`Q3 + 1.5·IQR`),
посчитанный по числу запросов среди повторных обращений (`count_rows >= 2`).
На обучающих данных это даёт `count_rows >= 5`. Порог пересчитывается из данных при
каждом запуске — конкретное число не зашито.

### 4. Финальное правило (И-условия)

Триггер аномалии срабатывает, если **одновременно**:

1. `score > 3.5` — OTS экстремально высок относительно бренда (стандартный порог
   выброса модифицированного z-score);
2. `count_rows >= порог активности` — активность действительно повторная;
3. `daily_ots > медианы OTS` — гарантия, что **малый OTS не может быть причиной**
   удаления 

## Почему именно так детектируется «аномально высокий OTS»

- Робастный z-score измеряет, насколько OTS респондента **выбивается вверх** из
  типичного распределения именно этого бренда: большой положительный `score` ⇔
  правый хвост ⇔ аномально высокий OTS. Малые/средние OTS дают `score ≈ 0` и не
  помечаются (плюс явный нижний порог по медиане).
- Порог `3.5` — общепринятый порог выброса, а не подгонка под целевой процент.
  Изменение порога — честный компромисс «снижение пиков ↔ потеря OTS», а не
  оптимизация под скрытую формулу. Число удаляемых респондентов нигде не задаётся
  вручную и не обрезается по дням — оно следует из общего правила.
- Условие активности отсекает ложные срабатывания «один запрос, но большой вес» и
  оставляет именно бот-подобное поведение (десятки запросов одного бренда за день).

## Влияние на отчётность (обучающие данные, июнь–октябрь 2025)

- Удалено уникальных респондентов: **123** (пар «респондент-день»: 169) из 23 379 —
  ≈ 0.53 %.
- Сохранено суммарного OTS: **≈ 96.5 %**.
- Снижение избытка экстремальных бренд-дневных пиков (выше 99.9-перцентиля): **≈ 36 %**;
  максимальный бренд-дневной OTS: 1 743 038 → 1 167 116.

Примеры аномалий (`anomaly_reasons.csv`): респондент ищет «honor» 10–13 раз в день
несколько дней подряд; респондент с «adidas» — до 36 запросов за день при весе
~44 000 (OTS ~1.6 млн за один бренд-день).

## Аналитические возможности 

Запускаются командой `python solution_Белоусов_Марк_ШЦТ-111.py --analytics`, результаты — в
`output/analytics/`. Все функции вызываются из кода без переписывания основной
логики; объекты для примеров выбираются из данных (топ-аномалия), без hardcode:

- `segment_before_after()` / `plot_segment()` — OTS до/после в разрезе:
  - демографии: `Пол`, `Возраст`, `Регион`, `Федеральный_округ`;
  - ресурсов: `ResourceName`, `ResourceType`, `Platform`, `UseType`;
  - уровней категорий: `CategoryDelivery`, `Category1`, `Category2`, `Category3`.
- `query_text_table()` — таблица `QueryText` выбранного аномального респондента за день.
- `plot_brand_ots_by_day()` — OTS по дням для выбранного бренда до/после очистки.
